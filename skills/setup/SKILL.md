# setup

Первый запуск свежего vault Огорода. Помогает пользователю стартовать и **выбрать
дашборд**, потому что их два (см. `dashboard/README.md`). Самодостаточно.

Это не про проект (для нового проекта — `charter`, для существующего кода —
`onboard`), а про **сам vault**: настроить и выбрать витрину.

## Когда запускать

- **Первый запуск:** в `projects/` только `example-project` (свежий клон шаблона) —
  предложи `setup` вместо `sync` (синхронизировать нечего).
- **По запросу:** «setup», «онбординг», «выбрать/переключить дашборд».

## Шаг 0 — прочитай правила

Прочитай `_rules.md` (методология) и `dashboard/README.md` (два уровня дашборда).

## Шаги

1. **Представь Огород** в двух фразах: персистентная память о проектах/задачах в
   markdown; ты (агент) ведёшь её, человек читает. Подробности — `_rules.md`.
2. **(Опц.) Obsidian + Dataview** — если человек хочет читать vault в Obsidian,
   подскажи поставить плагин Dataview (заработают сводки `_dashboard.md`/`tasks.md`).
   Не настаивай — это необязательно.
3. **Выбор дашборда.** Покажи два варианта и помоги решить по потребности:

   | | **mini** (`dashboard/serve.js`) | **full** (`dashboard-app/`, Next.js) |
   |---|---|---|
   | Запуск | `npm run dashboard` — без сборки | `npm install` + сервер |
   | Возможности | проекты · канбан · RICE · инбокс · аналитика · поиск | + Гант · роадмап · фазы · JTBD · документы · хостинг |
   | Кому | быстро посмотреть, локально | вести всерьёз, нужен хостинг и полный обзор |

   Спроси, какой выбрать (по умолчанию предложи **mini** — нулевой порог).
4. **Запиши выбор** в `artel.config.json` в корне vault: `{ "dashboard": "mini" }`
   или `{ "dashboard": "full" }`. Создай файл, если его нет. Это источник правды
   о выбранной витрине — читай его в начале сессии, чтобы знать, что запускать.
   Туда же запиши **версию шаблона**, с которой установлен vault — поле
   `ogorod_version` (= `tag_name` последнего GitHub-релиза Огорода без `v`,
   например `"1.1.0"`). По нему `sync` понимает, что вышло обновление. Итог:
   `{ "dashboard": "mini", "ogorod_version": "1.1.0" }`.
5. **Настрой выбранный:**
   - **mini:** запуск `npm run dashboard` (или `node dashboard/serve.js --open`). Зависимостей нет.
   - **full:** `cd dashboard-app && npm install`, затем `VAULT_PATH=<корень vault> npm run dev`
     (прод — `npm run build && npm start`). Переменные — в `dashboard-app/README.md`.
6. **Скажи про переключение:** в любой момент «переключи дашборд на full/mini» —
   обнови `artel.config.json` и настрой другой (шаг 5). Данные общие (один vault),
   переключение безопасно.
7. **Дальше:** удали `projects/example-project/`, заведи первый проект — `charter`
   (greenfield) или `onboard` (существующий код).

## Переключение (повтор по запросу)

«Переключи дашборд на <mini|full>» → перепиши поле `dashboard` в `artel.config.json`
и выполни настройку нужного уровня (шаг 5). Ничего не мигрируется — обе витрины
читают один markdown-vault.

## Самопроверка

- [ ] `artel.config.json` создан/обновлён с полями `dashboard` и `ogorod_version`
- [ ] Выбранный дашборд настроен и человек знает команду запуска
- [ ] Сказано, как переключиться позже
